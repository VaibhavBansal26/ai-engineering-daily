# 2026-07-03 — Kickoff: Building a Multi-Agent Content Automation System

## Concept

Today's entry is meta: the design of the system that maintains this repo. It's a practical example of **agent orchestration** — multiple specialized agents sharing state, with a human approval gate.

## Architecture

```
                 ┌────────────────┐
                 │ Topic Suggester │  (daily) researches trends,
                 └────────┬────────┘  dedupes against history
                          ▼
              ┌───────────────────────┐
              │  Notion: AI Content   │  shared state: topics, drafts,
              │       Pipeline        │  statuses, engagement metrics
              └──┬─────┬─────┬─────┬──┘
                 ▼     ▼     ▼     ▼
             GitHub  Medium LinkedIn YouTube    (specialist agents)
                 │     │     │     │
                 └─────┴──┬──┴─────┘
                          ▼
                 ┌─────────────────┐
                 │ HUMAN APPROVAL  │  nothing publishes without it
                 └────────┬────────┘
                          ▼
                      Published
                          │
                 ┌────────▼────────┐
                 │ Guardrails/Eval │  (weekly) scores quality,
                 │      Agent      │  flags violations
                 └─────────────────┘
```

## Why it matters

Agent systems fail in predictable ways: duplicated work, hallucinated facts, actions without oversight. The three mitigations here are shared state (one pipeline database instead of per-agent memory), a hard human-in-the-loop gate before any external action, and a separate evaluator agent — the agent that creates content never grades itself.

## Key takeaways

1. Orchestration is mostly a state-management problem — the database is the message bus.
2. Human-in-the-loop belongs at the *action* boundary (publish/commit), not the *thinking* boundary.
3. Separate evaluation from generation; self-graded homework is worthless.
